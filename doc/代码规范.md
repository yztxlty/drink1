# 喝水了吗 / Drink1 代码规范

> 适用范围：`app.js`、`pages/`、`components/`、`utils/`、`scripts/` 以及与其相关的资源文件。
>
> 本规范基于当前项目的真实结构编写，目标是让新代码保持一致、可维护、可回归。

## 1. 设计原则

- 本地优先：默认不依赖后端，业务数据优先保存在本机存储中。
- 单一数据源：饮水、设置、资料、勋章等核心状态统一由 `utils/store.js` 负责。
- 页面轻逻辑：页面负责交互、组装视图和路由，不承载复杂领域计算。
- 组件可复用：可跨页面复用的交互应抽到 `components/`。
- 文案统一：页面文案、按钮文案、提示语优先集中到 `utils/copy.js`。
- 兼容微信小程序运行环境：所有能力都要考虑 `wx` API 是否存在、是否可用。

## 2. 目录职责

### 2.1 `app.js`

- 负责初始化全局 store。
- 负责在启动时恢复本地状态。
- 负责将 `store`、`appState`、`userInfo` 暴露到 `globalData`。

### 2.2 `pages/`

- 每个目录对应一个页面或页面子页。
- 页面只负责：
  - 从 store 读取视图模型
  - 调用 store 的写接口
  - 响应用户操作
  - 负责跳转、弹窗、Toast、分享等页面级交互

### 2.3 `components/`

- 放可复用的 UI 组件和交互模块。
- 组件应尽量只通过 `properties` 接收输入，通过 `triggerEvent` 向外抛出事件。
- 组件不直接改写页面状态，不直接操作全局业务数据。

### 2.4 `utils/`

- 放纯逻辑、数据归一化、领域计算、文案和存储封装。
- `utils/store.js` 是核心状态层。
- `utils/copy.js` 集中管理产品文案。
- `utils/*.js` 中的领域工具应尽量无副作用。

### 2.5 `scripts/`

- 放回归校验、冒烟检查和开发辅助脚本。
- 当页面行为或跨页面联动发生变化时，优先补充对应校验脚本。

## 3. 命名规范

### 3.1 文件与目录

- 页面目录、组件目录优先使用小写命名。
- 多单词目录使用 `kebab-case`，例如 `data-management`、`quick-amount-manager`。
- 资源文件放在 `assets/`，文档截图放在 `docs/readme/`。

### 3.2 JavaScript 标识符

- 变量、函数、方法使用 `camelCase`。
- 常量使用 `UPPER_SNAKE_CASE`，例如 `COPY`、`MENU_ROUTES`、`SHARE_PATH`。
- 布尔值优先使用有语义的前缀：
  - `is...`
  - `has...`
  - `show...`
  - `visible...`
  - `can...`

### 3.3 方法命名

- 数据读取或刷新：`refresh...`、`load...`
- 数据构建：`build...`
- 数据解析或兜底：`normalize...`、`resolve...`
- 路由跳转：`go...`、`navigate...`
- 事件处理：`handle...`、`on...`

示例：

- `refreshPageData()`
- `buildShareTitle()`
- `normalizeQuickAmounts()`
- `goToMedals()`
- `handleQuickAmountAdd()`

## 4. JavaScript 编码风格

- 优先使用 `const`，仅在确实需要重新赋值时使用 `let`。
- 避免使用 `var`。
- 使用 CommonJS：`require(...)` / `module.exports`。
- 函数尽量短小，单个函数只做一件事。
- 优先使用早返回，减少嵌套层级。
- 对外部数据、存储数据、事件参数都要做容错处理。
- 访问 `wx` API 前先判断能力是否存在，避免在非标准环境或预览环境中报错。

### 4.1 分号风格

当前仓库历史文件中存在分号和无分号并存的情况。

- 新增文件时，请保持单个文件内部风格一致，不要混用。
- 修改旧文件时，优先沿用该文件已有的写法。
- 不要为了统一风格大范围重排无关代码。

### 4.2 注释使用

- 只给复杂、容易误解的逻辑加注释。
- 注释应解释“为什么这样做”，不要复述代码表面含义。
- 避免为简单赋值、简单条件分支写冗余注释。

## 5. 页面编写规范

### 5.1 页面数据

- `Page.data` 里只放当前页面渲染所需的数据。
- 初始值要完整，避免首屏空白。
- 视图模型字段应与 WXML 保持一致，避免“页面已经改了，模板还在读旧字段”。

### 5.2 页面生命周期

- `onLoad`：获取 `store`、初始化页面状态、做首次渲染。
- `onShow`：刷新依赖全局状态的数据，更新 TabBar 选中态。
- 页面切换回来时，不要假设 `data` 仍然是最新的。

### 5.3 数据读取

- 页面读取数据时，优先调用 `store.get*ViewModel()`。
- 页面不要自己拼装全局业务状态，尤其不要重复计算同一份统计口径。
- 派生数据应由 store 或纯工具层统一计算。

### 5.4 数据写入

- 页面修改业务数据时，必须通过 `store` 暴露的方法。
- 不要在页面里直接改写本地存储的业务对象。
- 对于仅影响局部体验的状态，可以留在页面内，但要和业务状态严格区分。

### 5.5 路由与反馈

- 页面跳转使用 `wx.navigateTo`、`wx.redirectTo`、`wx.reLaunch`、`wx.navigateBack`。
- 操作完成后及时给出 Toast、弹窗或状态反馈。
- 破坏性操作必须二次确认。

## 6. 组件编写规范

- 一个组件只聚焦一个能力，尽量避免“大而全”。
- `properties` 必须定义类型和默认值。
- 组件内部状态只保存和该组件自身交互有关的内容。
- 向外通信使用 `triggerEvent`，事件名应清晰直接。
- 组件里如果需要读取系统信息，应在 `lifetimes.attached` 或合适的生命周期内完成，并做好容错。
- 组件不要直接依赖页面私有状态。

### 6.1 组件事件命名

- 点击型事件优先用 `tap` 或明确的动词。
- 向外抛出的事件优先用动词命名，例如：
  - `select`
  - `change`
  - `delete`
  - `back`
  - `submit`

## 7. 状态与数据规范

### 7.1 `utils/store.js` 是唯一核心状态层

- 业务状态统一从 store 读写。
- 新增字段时必须同步处理：
  - 默认值
  - 归一化
  - 版本兼容
  - 视图模型输出

### 7.2 派生数据统一计算

- 连续打卡、累计记录、日统计、勋章进度等派生结果只保留一份计算口径。
- 不要在多个页面分别实现同一套统计逻辑。
- 如果统计口径变化，优先改 store 或 `utils/*.js`，再让页面跟着消费新结果。

### 7.3 存储键管理

- 固定存储键应集中管理，避免散落在页面里。
- 临时性、局部性的存储数据要明确命名和用途。
- 如果某个键只服务于页面交互，也要能清楚看出它的生命周期。

## 8. 文案与产品表达

- 产品语气保持温和、鼓励式、偏生活化。
- 术语尽量和项目现有表达一致，例如“补水”“记录”“森林”“勋章”“设置”。
- 统一使用 `utils/copy.js` 中的文案，减少页面内重复字符串。
- 新增文案时，要同时考虑：
  - 页面标题
  - 按钮文案
  - 空状态文案
  - 错误提示
  - 成功提示

## 9. 数据与隐私

- 默认遵循本地保存，不主动引入网络依赖。
- 涉及登录、资料、隐私说明或数据清空时，必须明确用户意图。
- 破坏性数据操作要有确认文案，并尽量说明影响范围。
- 如果未来接入云同步或账号系统，要同步更新隐私页、产品文案和相关校验脚本。

## 10. 测试与回归

- 改页面结构时，检查对应的 WXML / WXSS / JS 是否保持一致。
- 改核心逻辑时，优先补 `scripts/check-*.js` 中对应场景的校验。
- 影响首页、设置页、资料页、分享或数据管理时，建议至少跑一遍冒烟检查。
- 涉及跨页面联动的改动，要重点检查：
  - 首页记录
  - 设置修改
  - 资料同步
  - 勋章刷新
  - 分享内容

## 11. 提交前检查清单

- 页面新增字段是否都已在 `data` 中初始化。
- 业务状态是否全部经过 store 或领域工具处理。
- 文案是否已统一到 `utils/copy.js`。
- 组件是否只暴露必要的 `properties` 和事件。
- 是否补充或更新了对应的回归脚本。
- 是否避免了无意义的重复计算和重复存储。
- 是否保留了项目现有的本地优先与轻量结构。
