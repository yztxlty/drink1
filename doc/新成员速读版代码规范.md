# 喝水了吗 / Drink1 新成员速读版代码规范

> 这是一版给新成员快速上手的简版规范。先记住下面这些，开始写代码前再看完整版：[doc/代码规范.md](./代码规范.md)

## 一句话原则

- 这个项目是微信小程序原生写法。
- 业务数据默认本地保存。
- 页面负责展示和交互，`store` 负责核心状态，`utils` 负责纯逻辑，`components` 负责复用 UI。
- 文案尽量集中到 `utils/copy.js`。

## 先看这 5 条

1. 页面不要直接散写业务逻辑。
2. 记录、设置、资料、勋章这类状态优先走 `utils/store.js`。
3. 需要复用的交互，优先放到 `components/`。
4. 新增文案先看 `utils/copy.js`，不要在多个页面重复写同一句话。
5. 改动后顺手看一遍对应页面的 `scripts/check-*.js` 有没有相关校验。

## 目录怎么分工

- `app.js`：启动时初始化 store，挂到 `globalData`。
- `pages/`：页面渲染、交互、路由跳转。
- `components/`：可复用组件，比如导航栏、分享按钮、弹窗。
- `utils/`：日期、饮水统计、勋章、存储、文案等纯逻辑。
- `scripts/`：冒烟检查和回归脚本。

## 写代码时的习惯

- 函数和变量用 `camelCase`。
- 常量用大写下划线，例如 `COPY`、`SHARE_PATH`。
- 能用 `const` 就不用 `let`。
- 外部输入、存储数据、事件参数都要做兜底。
- 访问 `wx` API 前先确认它存在，别默认当前环境一定支持。

## 页面怎么写

- `onLoad` 负责拿 store、做首次渲染。
- `onShow` 负责刷新会变动的全局状态。
- `Page.data` 里只放当前页面要渲染的内容。
- 业务数据变化时，优先调用 store 提供的方法，不要直接改本地存储。
- 破坏性操作要二次确认。

## 组件怎么写

- 一个组件只做一件事。
- `properties` 要写类型和默认值。
- 组件内部只保留自己的局部状态。
- 对外发事件用 `triggerEvent`，事件名尽量直接。

## 最容易踩的坑

- 不要在多个页面各自实现一遍统计逻辑。
- 不要把文案散落在不同页面里。
- 不要直接改全局存储里的原始对象。
- 不要为了修一个页面去大范围重排无关代码。
- 不要忘记更新相关的回归脚本。

## 提交前快速自查

- 页面字段是否都初始化了。
- 新功能是否先过了 store 或工具层。
- 文案是否已经统一到 `utils/copy.js`。
- 组件事件和属性是否清晰。
- 相关脚本和文档是否一起更新。
