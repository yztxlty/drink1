# 森林互动模块重构设计规约

## 1. 目标

本次工作不是继续修补当前森林页的水滴实现，而是将 `pages/explore/explore` 的互动模块整体删除后重写，做成一个真正面向用户的“水滴花园”互动场景。

重构后的模块必须同时解决以下问题：

1. 视觉上要更像水，而不是一组渐变圆点。
2. 拖动过程要顺滑，不能因为频繁 `setData` 产生明显卡顿。
3. 页面切换后水滴位置、大小和当前场景状态不能被重新打散。
4. 达标或超标后必须进入“灌溉模式”，出现持续向下滴落的循环动画。
5. 互动区底部不再保留原来的地形、网格、雾层和重叠装饰，保持视觉干净。
6. 页面文案必须面向普通用户，移除“骨架”“预留”“后续扩展”等开发视角表述。

## 2. 核心决策

### 2.1 整体重写而不是增量修改

当前实现基于 WXML 中的多个 DOM 水滴节点，通过 `touchmove -> setData -> 重新布局` 完成交互。这种结构天然存在以下限制：

1. 水滴越多，拖动越容易卡顿。
2. 融合只能做“碰撞后直接替换”的硬切效果，难以表现液体粘连。
3. 页面重新显示时只能重新生成位置，无法自然恢复互动状态。
4. 灌溉模式下的持续动画会进一步放大 `setData` 开销。

因此本次重构直接删除旧实现，改为 Canvas 2D 驱动的独立互动引擎。页面保留现有森林页的数据概念，但互动区的渲染和动画逻辑全部重建。

### 2.2 模式划分

森林互动区只有两种正式模式：

1. 普通模式
   - 水滴静止分布在画布内。
   - 只有手动拖动才会触发融合。
   - 水滴之间不会自动靠近。

2. 灌溉模式
   - 触发条件：`todayIntake >= dailyTarget`
   - 未被手动拖动的水滴会缓慢相互吸附，靠近后自动融合。
   - 所有水滴持续向下滴落，到达底部后从画布上方重新生成。
   - 画布底部有合适数量的漂亮的树木或者鲜花。
   - 这是纯视觉循环动画，不减少水滴数量，不扣减饮水数据，不做消耗性结算。

## 3. 设计范围

本次重构包含以下内容：

1. 重写 `pages/explore/explore.js`
2. 重写 `pages/explore/explore.wxml`
3. 重写 `pages/explore/explore.wxss`
4. 重写 `utils/forest-water-rhythm.js`，让它从 DOM 布局辅助函数升级为 Canvas 水滴引擎工具
5. 扩展 `utils/store.js`，新增森林互动状态的持久化读写接口
6. 调整 `utils/copy.js` 中森林页相关文案
7. 增加或改写脚本校验，覆盖状态恢复、灌溉模式和用户文案

本次重构不包含：

1. 不新增后端或云存储
2. 不改动首页补水记录的数据口径
3. 不引入真物理引擎依赖
4. 不把灌溉模式做成积分消耗或能量结算系统
5. 不改动其他页面的逻辑和 UI 样式

## 4. 页面结构

### 4.1 保留的信息层

森林页仍然保留“补水成果”和“森林状态”的信息展示，但要减少对互动区的视觉侵入。

保留内容：

1. 顶部状态条
2. 森林概览数据卡
3. 互动区标题和简短说明
4. 底部结果或引导文案卡

### 4.2 互动区结构

互动区改成单一 `canvas`，不再把每颗水滴作为独立 WXML 节点渲染。

结构要求：

1. Canvas 外层为单一容器，用于测量尺寸和绑定触摸事件。
2. Canvas 内部只绘制水滴、液桥、高光、灌溉落点和轻量背景层。
3. 删除旧版底部地形、网格、雾层、重叠发光装饰。
4. 互动区底部绘制实体合适数量的漂亮的树木或者鲜花，保留简洁的水域空间感。

## 5. 视觉方向

### 5.1 整体气质

视觉关键词：

1. 清澈
2. 柔和
3. 流动
4. 安静
5. 有生命感

页面不应再呈现“游戏演示骨架”的感觉，而应更像一个可触碰的水滴花园。

### 5.2 水滴绘制

每颗水滴都使用 Canvas 的径向渐变绘制，至少包含三层信息：

1. 中心高光
   - 接近白色
   - 面积较小
   - 营造湿润与通透感

2. 主体颜色
   - 以湖蓝、浅青蓝为主
   - 保持一定透明度

3. 阴影和边缘
   - 外缘颜色更深
   - 底部带轻微阴影，让水滴更有厚度

必须提供独立的 `drawDrop(ctx, drop)` 绘图函数。

### 5.3 液桥效果

当两颗水滴接近但还未真正合并时，需要绘制“液桥”连接感，而不是只在距离达到阈值后瞬间替换。

视觉规则：

1. 只有距离足够近时才出现连接桥。
2. 液桥的透明度和宽度应随距离变化。
3. 普通模式下液桥主要在手动拖近时出现。
4. 灌溉模式下液桥也可以由自动吸附触发。

## 6. 数据模型

### 6.1 页面运行态

页面内维护的运行态至少包括：

1. `drops`
   - 每颗水滴的 `id`
   - 位置 `x/y`
   - 速度 `vx/vy`
   - 半径或尺寸
   - 色彩种子
   - 当前是否被拖动
   - 当前是否正在融合动画中

2. `interactionMode`
   - `normal`
   - `irrigation`

3. `activeDropId`

4. `canvasSize`

5. `animationState`
   - 上一帧时间戳
   - 当前是否在渲染循环中
   - 灌溉波动强度

### 6.2 Store 持久化状态

森林互动状态必须进入 `utils/store.js`，不能只存在页面内存里。

新增状态建议挂在 `business.forestScene` 或等价的业务区块下，至少包含：

1. `drops`
2. `mode`
3. `intakeSnapshot`
4. `targetSnapshot`
5. `updatedAt`

新增接口至少包括：

1. `getForestSceneState()`
2. `saveForestSceneState(sceneState)`
3. `clearForestSceneState()`

规则要求：

1. 每次拖拽结束后保存一次。
2. 每次融合完成后保存一次。
3. 页面 `onShow` 时优先从 Store 恢复。
4. 如果今日饮水量变化，只在现有状态基础上增减水滴，不整批随机重建。

## 7. 数据联动规则

### 7.1 水滴数量来源

水滴数量仍然由当日饮水量推导，但推导只是“数量基线”，不是“重新洗牌命令”。

具体规则：

1. 每 `50ml` 生成 1 颗基础水滴。
2. 页面首次进入且无缓存状态时，按当前饮水量生成初始水滴。
3. 页面再次进入时，优先恢复已有水滴位置。
4. 如果今日饮水量增加，则在现有场景末尾补充新水滴。
5. 如果今日饮水量减少，则从未拖动或尺寸最小的水滴开始回收，避免整个场景重排。

### 7.2 灌溉模式切换

切换条件：

1. `todayIntake < dailyTarget` 时为普通模式。
2. `todayIntake >= dailyTarget` 时为灌溉模式。

切换行为：

1. 普通模式切换到灌溉模式时，保留当前水滴布局。
2. 灌溉模式下只增加自动吸附和下落运动，不重置用户已摆放好的状态。
3. 再次进入页面时，如果饮水仍达标，直接恢复为灌溉模式。

## 8. 交互规则

### 8.1 拖动

拖动必须完全基于 Canvas 坐标系，而不是依赖 WXML 子节点坐标。

要求：

1. 使用 `wx.createSelectorQuery().select('#waterCanvas').fields({ node: true, size: true })` 获取 Canvas 节点和尺寸。
2. 记录设备像素比，对触摸点进行坐标换算。
3. `touchstart` 时通过命中测试找到最上层可拖动水滴。
4. `touchmove` 时只更新内存态和重绘，不在每一帧调用 `setData` 回写整组水滴。
5. `touchend` 时结束拖动并保存状态。

### 8.2 手动融合

普通模式下，只有被拖动的水滴靠近其他水滴时才触发融合。

融合规则：

1. 先进入液桥态。
2. 达到真正融合阈值后合并为更大的水滴。
3. 新水滴位置取两者中心加权结果。
4. 新水滴面积按两滴面积相加后换算半径，避免越融越假。

### 8.3 自动吸附

自动吸附只允许在灌溉模式下发生。

规则：

1. 非拖动状态的水滴会缓慢向最近邻移动。
2. 自动吸附的速度必须慢于手动拖拽，避免场景乱跳。
3. 若用户正在拖动某颗水滴，该水滴暂时不参与自动吸附。

### 8.4 灌溉循环动画

灌溉模式下，水滴会持续向下滴落。

规则：

1. 水滴受重力加速度影响，逐渐加快下落。
2. 到达画布底部后，从顶部重新生成。
3. 重生后的水平位置允许有轻微随机扰动，避免轨迹完全重合。
4. 这是无限循环视觉，不影响水滴总数和已记录饮水量。

## 9. 动画与渲染

### 9.1 主循环

必须采用基于 `requestAnimationFrame` 的刷新循环。

循环职责：

1. 更新拖动中的位置
2. 更新灌溉模式下的重力和速度
3. 更新自动吸附位移
4. 检查液桥和融合触发
5. 清空画布并重绘所有元素

### 9.2 重绘策略

页面数据层和画布层职责分开：

1. 页面文本、数字、状态文案仍通过 `setData` 更新。
2. 水滴位置、动画帧和拖动反馈只保留在内存和 Canvas 渲染层。
3. 避免每一帧把完整水滴数组同步进 `data`。

## 10. 文案方向

所有文案都必须改成面向用户的表达。

禁止出现的方向：

1. 骨架模式
2. 预留互动位
3. 后续扩展
4. 开发中
5. 规则占位

建议文案方向：

1. 页面标题：`水滴花园`
2. 页面副标题：`拖一拖，让水滴慢慢汇成一片清泉`
3. 空状态：`喝一点水，这里就会出现新的小水滴`
4. 达标提示：`今天的饮水目标已完成，水滴正在为森林持续灌溉`
5. 普通模式说明：`拖动水滴，靠近时它们会自然汇在一起`
6. 灌溉模式说明：`目标达成后，水滴会开始缓缓流动，给森林送去水分`

最终文案可以微调，但语气必须温和、直观、面向用户体验。

## 11. 重构边界

本次允许直接删除当前森林互动区旧实现，包括：

1. 旧的 DOM 水滴节点循环
2. 旧的拖动与融合逻辑
3. 旧的底部地形和装饰层
4. 旧的开发者视角说明文案

但以下内容应尽量复用：

1. 森林页的核心统计数据来源
2. 现有 `store` 的补水总量与目标读取方式
3. 共享 copy 管理模式
4. 现有音效资源

## 12. 验证要求

至少要覆盖以下验证点：

1. 首次进入页面时，水滴数量与今日饮水量一致。
2. 手动拖动时，水滴移动顺滑，坐标映射准确。
3. 手动拖近时出现液桥，达到阈值后融合成功。
4. 切换到其他 Tab 后再返回，水滴位置和大小保持不变。
5. 达标后自动进入灌溉模式。
6. 灌溉模式下出现持续下落和顶部重生的循环动画。
7. 灌溉模式下存在缓慢自动吸附，但普通模式下不存在。
8. 页面互动区没有旧版底部图层重叠问题。
9. 页面文案不再出现面向开发者的措辞。

## 13. 实施建议

为避免边改边乱，本次实现按以下顺序推进：

1. 先重构 Store 持久化接口
2. 再重写 Canvas 工具层和物理规则
3. 然后重写 `explore` 页面结构与样式
4. 最后做文案整理和回归校验

这次实现应视为“森林互动模块替换”，而不是“局部修补”。
