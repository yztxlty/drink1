# 补水档案中心重设计规约

## 1. 目标

将当前“我的”相关页面重构为一个统一的「补水档案中心」，让用户能够在一个清晰的入口里完成以下任务：

1. 查看个人基本信息。
2. 查看饮水连续性、总量、达标情况，以及分析报告和建议。
3. 查看系统级勋章总览与已获得勋章详情。
4. 初始化和调整补水目标、单次饮水量与快捷补水容量。
5. 导出标准 JSON 格式的补水资料，并展示存储路径。
6. 进入隐私条款、关于页并保持现状。
7. 退出当前微信账号登录，并支持登录后修改头像、昵称等信息。
8. 以更美观、更具收藏感的方式展示系统设置的勋章橱窗。

本次重设计的核心不是增加很多新页面，而是重新组织信息层级、视觉重心和数据联动逻辑，让页面更像一个“补水成长档案”而不是“设置列表”。

## 2. 设计原则

### 2.1 信息优先级

页面必须按以下优先级展示：

1. 当前身份状态与个人资料。
2. 当前补水进度与连续性。
3. 当前可执行操作，例如记录补水、调整目标、导出资料、查看勋章。
4. 次级功能，例如隐私条款、关于页、退出登录。

### 2.2 单一数据源

所有页面必须从同一份应用状态中读取和写回数据，避免以下问题：

- 首页记录的单次饮水量与设置页快捷容量不同步。
- 勋章页展示的数据与实际完成进度不一致。
- 导出数据与页面上看到的数据口径不同。

### 2.3 可解释性

页面上所有关键数字都要让用户知道它代表什么：

- 连续天数代表当前达标连续天数。
- 饮水总量代表累计补水总量。
- 分析报告要给出一句结论和一句建议。
- 勋章要显示“获得条件 + 当前进度 + 是否已解锁”。

### 2.4 低打扰、高激励

页面要有激励感，但不能像游戏任务面板一样过度复杂。做法是：

- 用视觉陈列强化勋章吸引力。
- 用分析总结强化用户对行为变化的感知。
- 用清晰入口减少找功能的成本。

## 3. 页面信息架构

### 3.1 主页：补水档案中心

主页是总览页，职责是“让用户一眼理解自己现在的补水状态”。

必须包含：

1. 个人资料卡
   - 微信头像或自定义头像。
   - 昵称。
   - 登录状态。
   - 编辑资料入口。

2. 核心指标卡
   - 连续达标天数。
   - 累计补水总量。
   - 已解锁勋章数或达标率。
   - 今日完成度。

3. 补水分析卡
   - 近 7 天完成情况。
   - 平均每日饮水量。
   - 平均单次饮水量。
   - 常见补水时段。
   - 一句话总结。
   - 一条建议。

4. 勋章预览卡
   - 展示最近解锁和即将达成的勋章。
   - 入口跳转到完整勋章橱窗。

5. 功能入口卡
   - 补水设置。
   - 补水勋章。
   - 同步补水资料。
   - 补水隐私条款。
   - 关于补水。
   - 退出登录。

### 3.2 勋章页：勋章橱窗与详情中心

勋章页负责展示系统勋章库和用户已获得的勋章状态。

必须包含：

1. 勋章总览头部
   - 已解锁数量。
   - 总勋章数量。
   - 完成率。
   - 当前成就鼓励文案。

2. 勋章橱窗展示区
   - 系统所有勋章卡片都可见。
   - 已解锁与未解锁视觉上区分明确。
   - 卡片美观，支持横向或网格式陈列。

3. 勋章详情态
   - 选中某个勋章后显示详细条件。
   - 展示当前进度、目标值、解锁时间。
   - 未解锁时显示“还差多少”。
   - 已解锁时显示“获得时间”和鼓励语。

### 3.3 设置页：补水控制台

设置页负责初始化和调整补水行为参数。

必须包含：

1. 每日补水目标设置。
2. 单次饮水量设置。
3. 快捷容量列表编辑。
4. 与首页记录逻辑互通的状态说明。

设置页的定位不是“系统设置”，而是“补水策略控制台”。

### 3.4 导出中心：同步补水资料

同步补水资料在产品定义上应被明确为“导出标准 JSON 文件”。

必须包含：

1. 导出按钮。
2. 导出格式说明。
3. 文件名规则。
4. 存储路径展示。
5. 用户后续可自行处理的说明。

### 3.5 隐私和关于页

保持现状，不改变功能范围，只做文案和视觉统一：

- 隐私条款继续保持本地优先说明。
- 关于页继续保留版本信息和反馈入口。

### 3.6 登录与资料编辑

登录方式仍然只有微信一键授权登录。

登录后支持：

- 修改头像。
- 修改昵称。
- 修改签名或个性描述。

用户手动编辑后的资料应优先于微信授权资料展示。

## 4. 视觉设计方向

### 4.1 主题

整体风格定义为「水感成长档案」。

关键词：

- 清透。
- 安静。
- 高级。
- 轻激励。
- 有收藏感。

### 4.2 色彩

建议采用以下层次：

- 主色：深海蓝或净水蓝。
- 辅色：青绿、浅蓝、白色。
- 背景：带微渐变的浅色背景，不使用纯白平铺。
- 卡片：半透明玻璃质感或柔和浮层。

禁止把页面做成普通灰白表单页。

### 4.3 排版

- 大数字用于连续天数、总量、完成率。
- 标题短、强、清楚。
- 说明文案尽量一屏内读完。
- 重要信息使用更高对比度，次要信息弱化。

### 4.4 勋章视觉

勋章橱窗应像“陈列柜”而不是“列表”：

- 已解锁勋章使用更高饱和度与光泽感。
- 未解锁勋章保留轮廓，但降低亮度和对比度。
- 勋章图标和名称不能互相挤压。
- 视觉上要有“想收集”的欲望。

## 5. 数据与联动逻辑

### 5.1 数据来源

所有页面都应基于同一套 store 数据，重点包括：

- `profile`
- `settings`
- `hydration.records`
- `hydration.daily`
- `hydration.streak`
- `hydration.totals`
- `achievements`

### 5.2 联动规则

1. 首页记录一次补水后，设置页中的单次容量和快捷容量应立即保持一致。
2. 勋章进度必须根据最新补水记录自动计算。
3. 主页的分析报告必须基于当前历史记录重新推导。
4. 导出 JSON 必须导出与 UI 相同口径的数据。
5. 退出登录不得删除本地补水记录和勋章成果，只清理当前账号态。

### 5.3 统计口径

必须统一以下统计定义：

- 连续天数：从今天往前连续达标的天数。
- 饮水总量：全部历史记录累计总量。
- 已解锁勋章：按勋章定义计算得出。
- 平均单次饮水量：总量 / 总记录数。
- 近 7 天完成情况：以最近 7 个自然日的达标情况统计。

## 6. 导出 JSON 规范

### 6.1 导出内容

建议导出以下结构：

```json
{
  "schemaVersion": 1,
  "exportedAt": "2026-03-31T00:00:00.000Z",
  "profile": {},
  "settings": {},
  "hydration": {
    "records": [],
    "daily": {},
    "streak": {},
    "totals": {}
  },
  "achievements": {},
  "meta": {}
}
```

### 6.2 导出命名

建议文件名：

- `drink1-export-YYYYMMDD-HHmmss.json`

### 6.3 路径展示

导出成功后，页面必须显示：

- 文件名。
- 保存位置。
- 可复制的路径或提示。

### 6.4 用户预期

文案必须明确告诉用户：

- 这是标准 JSON。
- 数据保存在手机本地存储中。
- 用户可以自行迁移、备份或清理。

## 7. 页面结构建议

### 7.1 主页

推荐结构顺序：

1. 顶部身份区。
2. 今日状态区。
3. 关键统计区。
4. 分析报告区。
5. 勋章预览区。
6. 功能入口区。

### 7.2 勋章页

推荐结构顺序：

1. 总览区。
2. 勋章橱窗区。
3. 勋章详情区。

### 7.3 设置页

推荐结构顺序：

1. 目标设置。
2. 单次饮水量与快捷容量。
3. 提醒策略。
4. 保存与说明。

## 8. 开发规范

### 8.1 代码边界

必须保持以下边界清晰：

- 页面负责展示和交互。
- `utils/store.js` 负责状态读写与聚合。
- `utils/water.js` 负责饮水统计计算。
- `utils/medals.js` 负责勋章定义和解锁评估。
- `utils/home.js` 负责首页展示模型。
- `utils/copy.js` 负责文案集中管理。

### 8.2 文案规范

全站文案应统一使用“补水”语境，避免同一概念多套说法。比如：

- 统一说“补水目标”。
- 统一说“补水资料”。
- 统一说“补水勋章”。

### 8.3 交互规范

- 任何保存类操作都必须给出明确反馈。
- 任何导出类操作都必须告诉用户导出结果。
- 任何跳转类操作都要有明确标题或副标题，不允许黑箱跳转。
- 退出登录前应做二次确认。

### 8.4 状态规范

- 页面首次进入时必须能展示默认态。
- 数据缺失时应显示合理兜底值，不能空白。
- 自定义头像和昵称优先于微信授权数据。

### 8.5 兼容规范

- 保留现有隐私页和关于页功能。
- 兼容旧存储字段。
- 避免因历史数据缺失导致页面崩溃。

## 9. 验收标准

以下条件必须全部满足：

1. 主页能清晰展示个人信息、连续性、总量、分析报告和建议。
2. 勋章页能看到系统所有勋章，并联动显示用户已获得的勋章详情。
3. 设置页可初始化和修改每日目标、单次容量与快捷容量。
4. 首页记录饮水与设置页单次容量互通。
5. 导出补水资料时生成标准 JSON，并显示文件路径。
6. 退出登录仅退出当前账号，不丢失本地补水成果。
7. 登录后可修改头像、昵称等个人资料。
8. 视觉上从普通设置页升级为更具收藏感和成长感的档案中心。

## 10. 风险与约束

### 10.1 平台导出能力

“写入手机存储卡并展示路径”在不同小程序能力环境下实现方式可能不同。设计上先把功能定义为“导出标准 JSON 并展示保存路径”，实现时根据平台能力选择具体落盘方式。

### 10.2 数据一致性

如果页面直接自己算统计，容易出现和 store 口径不一致的问题。因此统计必须尽量复用现有聚合逻辑，不能在页面里各算各的。

### 10.3 信息过载

主页内容较多，必须通过卡片分层、视觉层级和折叠/摘要策略控制密度，避免界面显得拥挤。

## 11. 推荐执行顺序

1. 先完成信息架构调整和文案统一。
2. 再完成主页与勋章页的视觉重构。
3. 然后完成设置页和导出中心。
4. 最后做隐私页、关于页和退出登录的细节统一。
